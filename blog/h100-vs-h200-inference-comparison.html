The H200 has 76% more VRAM (141GB vs. 80GB per GPU). The bandwidth improvement is 43% (4.8 TB/s vs. 3.35 TB/s). Those numbers look good in a spec sheet. The question is whether they matter for your workload.

Why Bandwidth Matters More Than Compute for Inference

This is the critical concept that most people miss.

In machine learning inference, you're not CPU-bound or compute-bound. You're memory-bandwidth-bound. Here's why: when a language model processes a token, it loads weights from VRAM into the GPU's caches, performs matrix multiplications, and writes results back. The multiplication itself is fast. The memory transfer is slow.

Consider V4-Flash with 13B active parameters. When you run a single forward pass, you load 13B parameters from VRAM. On H100 bandwidth (3.35 TB/s), that transfer takes roughly 4 milliseconds. The actual compute takes a fraction of that time. The bottleneck is not the math. It's the memory bus.

This is why larger batches don't help on constrained VRAM. A larger batch means more tokens, which means more independent inferences, which means you could parallelize them. But you can't use that parallelism because you still have to move all the model weights through the same memory bus. You're waiting on bandwidth, not compute.

The H200's 43% bandwidth increase is significant for this reason. It's the difference between 4ms and 2.8ms per token. That's 30% faster inference, all from bandwidth improvement.

The VRAM Difference: What Fits and What Doesn't

The H100 tier gives you 160GB. The H200 tier gives you 282GB. The question is simple: what models fit, and what doesn't?

V4-Flash at INT4 quantization fits on 160GB comfortably. So does Llama 3.1 70B in full precision (FP16). Mixtral 8x22B in quantization fits as well. You can load two models simultaneously on H100 if each is smaller than 80GB.

At 282GB, you gain flexibility. You can run larger models in less aggressive quantization. You can load multiple large models simultaneously. You can hold larger working datasets in VRAM. But "can" is different from "should."

The honest question: if your workload fits on 160GB, spending $275K more for 282GB VRAM is not a good investment. You're paying for capacity you won't use.

Who Should Buy H200 (Premium Tier)

H200 is for specific scenarios, and you should be honest about whether you have one.

You need H200 if: you're running models larger than 160GB, you need multiple large models loaded simultaneously, you're doing multimodal workloads (vision plus language), you have concurrent user load that requires higher throughput, or your inference pipeline includes external retrieval augmented generation (RAG) that requires holding large embedding datasets in VRAM.

Example: a large firm with 50 attorneys using the same system concurrently. You can't guarantee single-user performance. You need the bandwidth to serve multiple queries in parallel. The H200's 43% bandwidth advantage becomes material.

Another example: an organization running both V4-Flash and a vision model (when DeepSeek releases multimodal variants). V4-Flash at INT4 is ~142GB. A vision model might be 40-50GB. You'd want both in VRAM simultaneously. 160GB is tight. 282GB is comfortable.

But most organizations don't have these constraints. Most are running single models with predictable load. For those organizations, H200 is overprovisioning. The Premium tier ships Q3 2026 with $350K-$400K pricing. That's a premium you shouldn't pay unless you have a concrete reason.

The Performance Ceiling on H100

Let's talk about what you actually get in throughput terms.

On H100 with V4-Flash at INT4, you see 60-90 tokens per second for single-user inference (sustained). That's continuous inference without batching. If you batch multiple requests (accumulate several queries before processing), you can push 200+ tokens per second across all requests combined.

For a law firm with 30 lawyers, if each lawyer generates 10 queries per day, you're processing ~300 queries daily. At 500 tokens per query average, that's 150K tokens per day. H100 processes that in under a second of wall-clock time. You have massive headroom.

Even for 300 lawyers in a large firm, you're at 1.5M tokens per day. H100 processes that in 15-20 seconds. You don't need Premium tier for sequential workload.

You need Premium tier if you have concurrent load that demands parallel inference at scale. If 50 lawyers are querying simultaneously and you need <100ms response times, you need the bandwidth. If queries are sequential or loosely concurrent, H100 is fine.
