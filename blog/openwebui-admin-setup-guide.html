Why OpenWebUI Matters for Your Organization

If you've deployed Island Mountain hardware, you need a clean way to manage who accesses the AI models, what they can do with them, and how to keep records for compliance. OpenWebUI is the standard open-source interface for running multiple language models locally, and it handles user permissions, conversation history, and usage logging without sending anything to a cloud platform.

This guide covers the essential admin tasks: initial setup, user account management, role-based access to models, conversation history and retention, and network configuration for teams. We're assuming you've already deployed the hardware and have Ubuntu Server LTS running.

Initial Setup and Admin Login

OpenWebUI runs in Docker on your local network. After you've deployed the container (typically via our deployment automation), the first user created becomes the admin. You'll access it at http://ai.yourfirm.local:8080 or http://192.168.x.x:8080 depending on your network configuration.

When you first log in, you set an admin password. This is a critical security boundary. Use a strong password—20+ characters, mixed case, numbers, symbols. Store it in your password manager (Bitwarden, 1Password, or your firm's existing vault). This password controls access to user management, model configuration, and conversation history exports.

On first login, you'll see the admin panel in the top-right menu. From there, navigate to "Settings" to configure the basics:

  • Disable registration: Turn off public registration immediately. You want to control who gets accounts.
  • Enable authentication: Ensure login is required for all access.
  • Set conversation retention: We'll cover this in detail below.

Creating User Accounts and Role Assignment

OpenWebUI separates users into three basic roles: Admin, User, and Moderator. For most organizations, you'll only use Admin and User. The difference is simple: Admins see the admin panel and can modify settings; Users can access models and create conversations but can't change global configuration.

To create a user account, go to Admin Settings > User Management. Click "Add User." Set the username (usually firstname.lastname or email), password, and role. The password can be temporary—users can reset it on first login if you enable that option.

For law firms, you might have roles like "Attorney," "Paralegal," and "Staff." OpenWebUI doesn't enforce role-based permissions at the model level natively, but we'll address that below. Start by creating accounts for your team members, then restricting what models they can access.

Model Access Control by Role

Here's where it gets specific to your workflow. You've deployed multiple models: DeepSeek V4-Flash (quantized), Llama 3.1 70B, and Mixtral 8x22B. In a law firm, you might want:

  • Attorneys and senior paralegals: Access to all models, including V4-Flash for fast draft analysis
  • Junior paralegals: Access to Llama 70B and Mixtral, but not V4-Flash (cost control)
  • Administrative staff: Limited to Llama 70B for document summarization only

OpenWebUI handles this through the "Models" section in Admin Settings. Each model has an "Allow" list. You can restrict which users or groups can access it. Currently, OpenWebUI doesn't have built-in group management—you manage access per user. This means for a 50-person organization, you'll be managing model access individually.

A workaround: create a simple spreadsheet tracking which users should access which models, then batch-update OpenWebUI settings monthly. This is more manual than enterprise IAM, but it works for organizations under 200 users and doesn't require additional software.

Conversation History, Retention, and Compliance

OpenWebUI stores all conversations locally on your server. This is a major compliance advantage: your conversation data never leaves your network. But it's also a responsibility: you need to manage retention, backup, and access.

Local storage: All conversation data is stored in a SQLite database on your server. The database location is typically /root/ollama/web/conversations.db depending on your deployment. This is local to your server, not in cloud storage.

Admin visibility: In Admin Settings, there's a "Conversations" tab. As admin, you can see all conversations across all users, search by date or content, and export them. This is important for compliance: if you're subject to discovery requests in litigation or regulatory audit, you have full access to all model interactions.

Retention policy: Set a retention window. Most organizations delete conversations older than 12 months automatically. Go to Admin Settings > System > Conversation Retention and set the number of days. Common settings: 365 days for law firms (to maintain litigation hold capability), 90 days for IT support teams, 180 days for medical practices.

JSON export: You can export conversations as JSON for archival. This is useful if you're migrating data, conducting a full backup, or need to migrate to a different system. The export includes timestamps, model used, user, and full conversation text.

For HIPAA-regulated organizations (medical practices, covered entities), remember that conversations might contain protected health information. Your retention policy, backup procedure, and access controls all need to satisfy HIPAA requirements. Consider encrypting the database file at rest and ensuring only authorized personnel can access the admin panel.

Usage Monitoring and Analytics

OpenWebUI logs basic usage: which user ran which model and when. This data is stored locally and accessible from the admin panel under "Analytics" (if enabled). You can see per-user token counts, model usage over time, and identify power users.

This isn't as detailed as enterprise AI audit logging, but it's sufficient to track adoption, identify training needs, and catch unusual activity (e.g., a user suddenly running hundreds of queries per day, which might indicate they're using the API programmatically).

For cost allocation across departments, you can use this data to track which teams are using the models most. Some firms bill AI usage back to practice areas based on this data.

Network Configuration: Static IP and Hostname

Your Island Mountain server needs a stable network location. Users shouldn't have to remember an IP address.

Static IP: On your network router or DHCP server, assign a static IP to the Island Mountain server. Something like 192.168.1.50 works. This prevents the server from getting a new IP address when it reboots.

Hostname and DNS: Give the server a meaningful hostname like ai.yourfirm.local or gpt-local.yourfirm.local. In your internal DNS or hosts file, point this hostname to the static IP. Now users can access OpenWebUI at http://ai.yourfirm.local:8080 instead of remembering IP addresses.

If you have a Windows domain (Active Directory), you can register the hostname automatically. If not, add it to your hosts file on each user machine or configure it in your router's DNS settings.

Security: VPN and Network Isolation

Your Island Mountain server should not be directly exposed to the internet. Ideally, it's only accessible from your internal network. If you have remote staff, use a VPN.

Tailscale: Install Tailscale on your server and all user machines. Tailscale creates an encrypted mesh network where only devices in your Tailscale account can access each other. Users who are working from home or traveling can access the OpenWebUI interface the same way as on-site staff. Tailscale is free for personal use and cheap for small teams (~$10/user/month for business use).

WireGuard: If you prefer open-source and self-hosted, deploy WireGuard on your server. WireGuard is lighter than traditional VPN protocols and handles remote access without cloud intermediaries. It requires more configuration than Tailscale, but gives you complete control.

Either way: your Island Mountain server should not be accessible directly from the internet. All remote access goes through your VPN tunnel.

Backup Strategy: Database and Data Retention

You need a backup of the OpenWebUI database. Use these practices:

Local backup: On the Island Mountain server, run a daily backup of the OpenWebUI database to a local NAS or external drive. A simple cron job works: sqlite3 /root/ollama/web/conversations.db ".backup /mnt/nas/backup/conversations-$(date +\%Y\%m\%d).db" daily at 3 AM.

NAS, not cloud: Store backups on your local network-attached storage (NAS) like a Synology or Buffalo box. Do not sync these backups to AWS S3, Google Drive, or Dropbox. You're using on-premises hardware specifically to keep data off cloud platforms. Syncing backups defeats the point.

Retention: Keep 30 days of daily backups. After 30 days, delete old backups. This balances recovery capability against storage cost.

Encryption: If your NAS supports encrypted shares, enable it. OpenWebUI database files may contain sensitive information—they should be encrypted at rest.

Common Admin Tasks

Adding a new model: If you've added another model to your Ollama installation, you need to enable it in OpenWebUI. Go to Admin Settings > Models and refresh the model list. The new model should appear. Set permissions as needed.

Resetting a user password: You can reset any user's password from the User Management panel. The user will be logged out and forced to set a new password on next login.

Viewing a specific conversation: Go to Admin Settings > Conversations, search by user or date, and click the conversation to view it. This is critical for compliance—you have full audit access.

Exporting all conversations: Some organizations do quarterly exports of all conversations to an archive for long-term retention. Go to Admin Settings and look for the export option. This generates a large JSON file with all conversations and metadata.
